Resumen Ejecutivo

La aplicación ipc_app.exe es una sofisticada demostración de Comunicación Inter-Procesos (IPC) a nivel de sistema operativo en entornos Windows. Su objetivo principal es ilustrar cómo dos procesos independientes (un Padre y un Hijo) con espacios de memoria aislados pueden coordinarse e intercambiar datos de forma segura y eficiente.

A diferencia de sistemas básicos, esta aplicación implementa un enfoque dual, contrastando dos paradigmas fundamentales del OS: Tuberías Anónimas (Pipes) mediadas por el Kernel para el intercambio sincronizado, y Memoria Compartida con Nombre (Named Shared Memory) para transferencias de ultra-baja latencia mapeadas directamente en la RAM.

Arquitectura del Sistema

El modelo se basa en una jerarquía Padre-Hijo. El proceso Padre actúa como orquestador: configura los canales de comunicación, reserva los bloques de memoria física y despacha al proceso Hijo delegados, transmitiendo los manejadores (Handles) necesarios como argumentos de línea de comandos.

Flujo de Ejecución Profundo

A continuación se detalla la secuencia algorítmica y las intervenciones del Kernel durante la ejecución, monitoreando paso a paso el ciclo de vida del IPC.

1. Creación de Canales (Modo Kernel)

El proceso Padre invoca la Win32 API para establecer los vectores de comunicación.

  • Pipes: Se instancian dos pares de pipes (Lectura/Escritura) asegurando que los atributos de seguridad SECURITY_ATTRIBUTES permitan la herencia de handles (bInheritHandle = TRUE).
  • Object Manager: El núcleo intercepta CreateFileMappingA. Se crea un objeto de tipo "Section" referenciado en el namespace global como Local\PruebaMemoriaCompartida_P6, delegando 1024 bytes de páginas virtuales apoyadas por el archivo de paginación del SO.

2. Despacho del Proceso Hijo

El padre serializa los IDs de los Handles requeridos (representados como enteros de 64 bits) en la línea de comandos y ejecuta la syscall encargada de bifurcar procesos.

std::string cmd = std::string(exePath) + " child " +

CreateProcessA(NULL, (LPSTR)cmd.c_str(), NULL, NULL, TRUE, ...);

Operación del SO: El Memory Manager establece un nuevo contexto VAD (Virtual Address Descriptor). Se asigna un PID único. El Object Manager copia las entradas marcadas como heredables de la Tabla de Handles del Padre a la del Hijo.

3. Eventos de Cambio de Contexto (Context Switch)

La aplicación realiza un seguimiento proactivo de los cambios de contexto del CPU. Cuando el Hijo ejecuta un ReadFile() bloqueante en el Pipe vacío, ocurre un Context Switch crítico:

Razón de interrupción: Solicitud I/O Bloqueante.
Acción de hardware/software: El Kernel guarda los registros actuales (EIP, ESP, EAX, etc.) en el Thread Control Block (TCB) del flujo hijo. El estado del hilo muta de RUNNING a WAITING. El Planificador (Scheduler) asigna el ciclo de reloj libre al proceso Padre o a otro proceso READY de mayor prioridad en la cola del SO.

4. Intercambio por Memoria Compartida

A diferencia de la Tubería, la validación del objeto de memoria es delegada sin la sobrecarga del paso de mensajes continuo. El proceso Padre pre-carga en la RAM el dato ("Datos secretos en RAM...").

El proceso Hijo emplea OpenFileMappingA apuntando al mismo Namespace. Tras una validación por el Security Reference Monitor (SRM), MapViewOfFile() asigna las páginas físicas involucradas a direcciones virtuales continuas en el User Space del hijo.

Cero Syscalls: El Hijo lee, procesa y responde sobrescribiendo la misma dirección de memoria usando operaciones de puntero clásicas (memcpy), operando a velocidad nativa de la memoria subyacente de la arquitectura x86_64.

5. Limpieza y Terminación

Un aspecto imperativo registrado por el app es la prevención de fugas de recursos (Resource Leaks). Ambos procesos invocan UnmapViewOfFile retirando las páginas mapeadas de su tabla de páginas, seguido de sucesivas llamadas a CloseHandle.

Cuando el contador de referencias interno del Object Manager para cada entidad (Section, Pipe, Process, Thread) alcanza cero, los destructores en el espacio del Kernel reclaman las estructuras subyacentes físicamente.
